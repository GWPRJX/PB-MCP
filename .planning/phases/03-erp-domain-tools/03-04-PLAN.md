**Step 2: Fix rls-isolation.test.ts**

This test seeds cross-tenant data and verifies isolation. It fails because Phase 3 tables (products, stock_levels, orders, order_line_items, invoices, contacts) are not seeded in its beforeAll. Fix by adding seed rows for Phase 3 tables in the test's beforeAll, using the same superuser DATABASE_MIGRATION_URL pattern. Each table needs at least one row per tenant so the isolation assertion (Tenant A cannot see Tenant B's rows) is exercised.

Minimum seed additions for each tenant in beforeAll:
- `contacts`: INSERT one row per tenant
- `products`: INSERT one row per tenant
- `stock_levels`: INSERT one row per tenant (linked to product)
- `orders`: INSERT one row per tenant (linked to contact)
- `order_line_items`: INSERT one row per tenant (linked to order + product)
- `invoices`: INSERT one row per tenant (linked to contact)

The teardown (afterAll) already deletes tenants via CASCADE — no additional cleanup needed.

**Step 3: Fix rls-policy-coverage.test.ts**

This test checks that every tenant-bearing table has at least one RLS policy. Phase 3 tables already have policies (created in Phase 1 SQL migrations). This test likely fails because its list of expected tables did not include the Phase 3 tables, not because the policies are missing. Read the test to understand:
- If it hardcodes a table list: add the 7 Phase 3 tables to the expected list
- If it queries pg_policies dynamically: the failure is elsewhere — read the actual error output

Do NOT add new migrations or alter the schema. The policies exist; only the test expectation needs updating.

**Step 4: Fix app-role.test.ts**

The 42P07 vs 42501 mismatch means a test expects error code 42501 (insufficient_privilege) but receives 42P07 (duplicate_table) or vice versa. Read the test and the actual error output carefully.

Common cause: a test tries to CREATE TABLE as the app role and expects to fail with 42501 (no CREATE privilege), but if the table already exists from a prior test run the error is 42P07 (duplicate). Fix by either:
- Using `CREATE TABLE IF NOT EXISTS` so the duplicate error never fires, then asserting the app role got the object without privilege
- Or catching both error codes as acceptable in the assertion
- Or adding a DROP TABLE IF EXISTS in beforeAll/afterAll cleanup

Read the test first before deciding. Do NOT change the database schema or roles.
