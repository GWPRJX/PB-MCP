  1. **tests/admin/tool-permissions.test.ts** — Integration tests for TAC-01, TAC-02, TAC-03.

     Setup: Create a test tenant via `createTenant()` in `beforeAll`. Clean up in `afterAll` (delete tenant via direct SQL using DATABASE_MIGRATION_URL). Set process.env.DATABASE_MIGRATION_URL if needed.

     Test cases:
     - `getToolPermissions returns all 21 tools defaulting to enabled` — Call `getToolPermissions(tenantId)`, expect 21 items, all with `enabled: true`.
     - `updateToolPermissions disables specific tools` — Call `updateToolPermissions(tenantId, [{ toolName: 'list_products', enabled: false }])`. Then `getToolPermissions` and verify list_products is disabled, others remain enabled.
     - `updateToolPermissions re-enables a disabled tool` — Disable then re-enable a tool. Verify it's enabled again.
     - `getEnabledTools returns only enabled tools` — Disable 2 tools, call `getEnabledTools(tenantId, null)`, verify the disabled tools are NOT in the result and the count is 19.
     - `getEnabledTools with key allowedTools returns intersection` — Enable all 21 tools at tenant level. Call `getEnabledTools(tenantId, ['list_products', 'get_product'])`. Expect only those 2 tools returned.
     - `getEnabledTools with key allowedTools AND tenant disabled` — Disable 'list_products' at tenant level. Call `getEnabledTools(tenantId, ['list_products', 'get_product'])`. Expect only 'get_product' (intersection excludes tenant-disabled tool).
     - `updateKeyAllowedTools sets per-key restrictions` — Create an API key, call `updateKeyAllowedTools(tenantId, keyId, ['list_products'])`, verify returns true.
     - `updateKeyAllowedTools with null clears restrictions` — Call `updateKeyAllowedTools(tenantId, keyId, null)`, verify returns true.
     - `updateKeyAllowedTools returns false for non-existent key` — Call with random UUID, expect false.

  2. **tests/admin/audit-log.test.ts** — Integration tests for TAC-04, TAC-05.

     Setup: Create a test tenant and API key in `beforeAll`. Clean up in `afterAll`.

     Test cases:
     - `recordToolCall creates an audit log entry` — Call `recordToolCall(tenantId, keyId, 'list_products', { limit: 10 }, 'success', undefined, 42)`. Then `queryAuditLog(tenantId)` and verify the entry exists with correct fields.
     - `recordToolCall records error entries` — Call `recordToolCall(tenantId, keyId, 'get_product', { id: '999' }, 'error', 'Not found', 15)`. Query and verify status='error' and errorMessage='Not found'.
     - `queryAuditLog filters by toolName` — Record calls for 'list_products' and 'get_product'. Query with `{ toolName: 'list_products' }`. Verify only list_products entries returned.
     - `queryAuditLog filters by status` — Record success and error calls. Query with `{ status: 'error' }`. Verify only error entries returned.
     - `queryAuditLog filters by both toolName and status` — Query with both filters, verify correct subset.
     - `queryAuditLog supports pagination` — Record 5 entries. Query with `{ limit: 2, offset: 0 }`. Verify 2 entries returned and totalCount is 5+. Query with `{ offset: 2, limit: 2 }` for next page.
     - `recordToolCall is fire-and-forget (does not throw on DB errors)` — This is hard to test directly. Instead, verify that calling recordToolCall with a non-existent tenantId does NOT throw (it logs to stderr internally). Wrap in a try/catch and assert no error thrown.

<task type="auto">
  <name>Task 2: JWT auth and API key expiry integration tests</name>
  <files>tests/admin/jwt-auth.test.ts, tests/admin/key-expiry.test.ts</files>
  <action>
  1. **tests/admin/jwt-auth.test.ts** — Integration tests for AUTH-01.

     Setup: Set `process.env.JWT_SECRET = 'test-jwt-secret'` and `process.env.ADMIN_SECRET = 'test-admin-secret'` in `beforeAll` (or ensure they're set in test environment).

     Test cases:
     - `signJwt creates a valid JWT string` — Call `signJwt({ sub: 'admin' })`, verify it's a string with 3 dot-separated base64url parts.
     - `verifyJwt accepts a valid token` — Sign a token, then verify it. Expect decoded payload with `sub: 'admin'`, `iat`, and `exp` fields.
     - `verifyJwt rejects a token with wrong signature` — Sign a token, tamper with the signature part, verify returns null.
     - `verifyJwt rejects an expired token` — Sign a token with `exp` set to a past timestamp (manually construct or temporarily set JWT_EXPIRY_HOURS to a very small value). Verify returns null.
     - `verifyJwt rejects malformed tokens` — Pass empty string, 'not.a.token', and 'a.b' (2 parts). All should return null.
     - `POST /admin/auth/login returns JWT on valid credentials` — Use Fastify's `inject` method (import `buildServer` from `../../src/server.js`). Build server, inject POST /admin/auth/login with correct username/password. Expect 200 with `{ token: string }`. Verify the returned token is valid via `verifyJwt`.
     - `POST /admin/auth/login returns 401 on invalid credentials` — Inject with wrong password. Expect 401.
     - `Protected admin route accepts JWT Bearer token` — Sign a JWT, inject GET /admin/tenants with `Authorization: Bearer <token>`. Expect 200.
     - `Protected admin route accepts X-Admin-Secret header` — Inject GET /admin/tenants with `X-Admin-Secret: <secret>`. Expect 200 (backward compatibility).
     - `Protected admin route rejects invalid JWT` — Inject with `Authorization: Bearer invalid-token`. Expect 401.
     - `Protected admin route rejects missing auth` — Inject with no auth headers. Expect 401.

     Note: Use `server.inject()` for HTTP-level tests (Fastify's built-in test helper — no real HTTP server needed). Call `server.close()` in afterAll.

  2. **tests/admin/key-expiry.test.ts** — Integration tests for AUTH-02.

     Setup: Create a test tenant in `beforeAll`. Clean up in `afterAll`.

     Test cases:
     - `createApiKey without expiresAt creates a key that never expires` — Call `createApiKey(tenantId)`, lookup the key hash, verify result has tenantId (not expired, not null).
     - `createApiKey with future expiresAt creates a valid key` — Call `createApiKey(tenantId, 'test', new Date(Date.now() + 86400000).toISOString())` (1 day in future). Lookup by hash, verify it returns the normal result (not expired).
     - `createApiKey with past expiresAt creates an expired key` — Call `createApiKey(tenantId, 'expired', new Date(Date.now() - 86400000).toISOString())` (1 day in past). Lookup by hash. Expect the result to indicate expired: check for `{ expired: true }`.
     - `lookupApiKeyByHash returns null for non-existent hash` — Lookup a random hash. Expect null.
     - `lookupApiKeyByHash returns null for revoked key` — Create a key, revoke it, lookup by hash. Expect null.

     Note: The `createApiKey` function returns `{ apiKey, rawKey }`. Use `generateApiKey()` to get the hash, or compute `createHash('sha256').update(rawKey).digest('hex')` for lookup testing.
